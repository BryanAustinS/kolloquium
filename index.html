Current Practice: Hand-Written Rules

A hand-written Groovy rule engine inside the ETL pipeline — checking only the fuel columns, e.g. the mass balance |block − (remaining + uplift)| ≤ 200 kg. All other operational columns pass through unvalidated.
Brittle
Every new exception triggers another if–else branch.
Expensive
Every rule change requires a new software release.
Incomplete
Anomalies nobody anticipated pass through silently.
A trip fuel of 10,000 kg is a perfectly valid number —
but implausible for an A320 on a 30-minute leg. No schema or type check will ever catch it.

Research Questions

RQ1
To what extent do the results of a rule-based plausibility check differ from those of an ML-based classifier — applied to the same flight data?
RQ2
Can the ML classifier replace the classical rule-based logic — or does it only complement it?
plus an engineering goal: a configurable web application where domain experts
run both approaches on their own data and extend the rule set — without touching production code

Stage 1 — How a Record Gets Its Label

1 · Compare like with like. A cluster groups comparable flights, e.g. same airline, year, aircraft type & route:
LH | 2026 | A320 | FRA | AMS
2 · Flag statistical outliers per group & column with the IQR fence:
[Q1 − 1.5·IQR,  Q3 + 1.5·IQR]
robust — outliers barely move the quartiles (chosen over 3σ with the domain experts)
3 · Explain what an expert recognises. Three expert rules from Lufthansa Systems:
Low Cargo (frequent) · Ferry flight (moderate) · ZFW OK – No Uplift (rare)
three real flights, route SN A320 BRU→GVA — take-off-weight bounds [51,653 – 71,219 kg]
Revenue flight · TOW 57,547 — inside bounds clean
Ferry flight · TOW 48,115 — flagged, but ferries fly empty:
rule Service_Type_Ferry fires explained
Revenue flight · TOW 50,805 — flagged,
no rule accounts for it anomaly
The flag alone does not decide — the rule engine does.
Unexplained anomalies are the detection target of the whole system.

Stage 2 — Learning Setup

Supervised multiclass classification — one of five classes per record: clean · 3 rule classes · anomaly (the target)
Dummy classifier
always predicts "clean" — the no-skill floor any real model must clearly beat
Logistic Regression
linear, interpretable baseline — cannot capture feature combinations
LightGBM
gradient-boosted trees — the standard for large tabular data; learns interactions
Features: ~200 raw columns → 100 used
39 raw numeric · 50 engineered domain features · 11 one-hot service types
fuel per distance · OFP-vs-loadsheet residuals · physical identities like TOW−(ZFW+fuel) · actual-vs-scheduled times · detour ratio ground/great-circle
Selection & tracking
kept = union of each model's top-30 by feature importance (~90% of total gain)
every run logged with MLflow — reproducible & traceable
a large or unexpected value in any engineered feature is a direct signal that something does not fit together

Training for Rare Anomalies

Class imbalance → undersampling
clean majority trimmed to 10× the largest minority class, only inside CV training folds.
class weights → precision collapsed · SMOTE → synthetic noise, no gain
Hyperparameter search
Logistic Regression: grid search
LightGBM: Optuna — 450 trees, 117 leaves, depth 10, lr 0.163
Decision threshold 0.9 — precision first
A miss is cheap: retrospective data-quality check on flights already flown — no safety impact, catchable later.
A false alarm is expensive: every flag is reviewed by hand — noise erodes the analysts' trust.
Headline metric: PR-AUC
With 2.7% positives, accuracy and ROC-AUC mislead.
No-skill PR-AUC = the anomaly rate ≈ 0.027.
evaluated on a 30% held-out test set (in-distribution) and on the separate unseen-routes file

Choosing the Operating Point

precision · recall · F1 vs decision threshold — anomaly class
Missed anomaly → cheap.
Retrospective check on flights already flown; a later run can still catch it.
False alarm → expensive.
Every flag is reviewed by hand — noise swamps analysts and costs trust.
At 0.9: ~¾ of flags are real — a small, credible review queue.
full sweep stored → operating point movable without retraining

RQ1 — How Far Do They Differ?

On the unseen file the two verdicts agree on 96.6% of 149,176 records — the disagreements are the interesting part.
They agree where patterns are common & clear
clean F1 0.98 · Low Cargo 0.67 · Ferry 0.67
for frequent, rule-shaped patterns the model reproduces the rules almost exactly
They diverge on the rare & the unexplained
hard yes/no + explanation vs probability over many columns —
the model softens rigid thresholds and flags near-misses; on the scarce class the rule alone is right
Model flags, no rule does: 865 records (0.6%)
98% land on records the rules call clean → candidate anomalies in the majority no rule ever inspects
Rules flag, model misses: 4,254 records (2.9%)
the precision-first threshold's low recall, seen from the other side
the difference is not one number — it is a function of how common and how rule-shaped a pattern is:
near-zero on frequent classes · near-total on rare ones

RQ2 — Replace or Complement?

The evidence points to complement — replacement is not supported at the current data size.
1 · Ceiling = rule mimicry. Trained on rule labels, it cannot define anomalies by itself.
2 · Recall 24%. Used alone, roughly three in four anomalies slip through.
3 · Scarce patterns unlearnable. ZFW OK: F1 0.04 vs a rule that fires every time.
4 · Degrades off-distribution. 0.49 → 0.30 PR-AUC on unseen routes.
The converse also fails: rules are rigid, single-column, and miss soft multivariate cases — each covers what the other cannot → keep both.
Still open: are the model's 865 extra flags real anomalies? The rule-label metrics cannot say — a blind expert review at Lufthansa Systems (~6,100 disagreement records, models anonymised as "Model 1 / Model 2", IQR bounds shown as evidence) will settle it.

Limitations & Future Work

Limitations
  • single export · one airline group · three rules
  • labels cap the model — it cannot discover a genuinely new type of anomaly
  • rule-label metric is one-sided: extra flags count as false alarms even if real
  • random split, not temporal — drift unmeasured
  • data-limited: cross-file score had not plateaued
Future work
  • unsupervised / semi-supervised detection — beyond the rule-label ceiling
  • more data → scarce classes become learnable
  • grow the rule set → richer, more balanced labels for Module B
  • per-rule thresholds & probability calibration
  • temporal validation · complete the expert review
read the reported numbers as evidence for the approach, not as final performance figures

Conclusion

Architecture
Two-stage hybrid design — statistical detector + rule engine feeding a supervised classifier; rules & clusters configurable at runtime.
Empirical comparison
First rule-vs-ML comparison on the same real tabular flight data: LightGBM 18× no-skill, precision 0.76, recall 0.24 — and where each approach wins.
Working artefact
Streamlit web application running both stages on any numeric column of an uploaded CSV — in use for the expert evaluation.
Rules and machine learning are not rivals —
they are two stages of the same pipeline.
The rules stay the trusted, explainable core that experts read and edit.
The model adds a fast layer that surfaces candidate anomalies in the 95% clean majority no rule ever inspects.
